# mp3-player-embedded
An mp3 player made for a prototype board
## Setup

1. Connect the board to your computer using a USB cable.
2. Open VS Code and connect to the board using MicroPico.
3. Upload both files to the board.
4. Run `audio_player_frontend.py`.

## Hardware Connections

| Signal                | GPIO |
| --------------------- | ---- |
| OLED SDA              | 12   |
| OLED SCL              | 13   |
| Encoder A (ANG_A)     | 18   |
| Encoder B (ANG_B)     | 17   |
| Encoder Button (SW_C) | 5    |
| Mode Button (BTN)     | 21   |

## How to Use It

### Volume/Play Mode

* Turn the encoder to change the volume (0–30).
* Press the encoder to play or pause the song.
* Press the mode button to switch to Song Select mode.

### Song Select Mode

* Turn the encoder to move through the song list.
* Press the encoder to select and play a song.
* Press the mode button to go back to Volume/Play mode.

## Adding Songs

To add songs, update the `song_list` in `audio_player_frontend.py` so it matches the files on your SD card. The DFPlayer Mini expects song names like:

* `0001.mp3`
* `0002.mp3`
* `0003.mp3`

and so on.

# mp3-player-embedded

A standalone digital audio player built on the YD-RP2040 microcontroller,
running MicroPython. It streams CD-quality WAV audio from a microSD card to an
external I2S DAC in real time, with a rotary-encoder-and-OLED user interface.
Audio playback is decoupled from the interface so the controls stay responsive
while music plays.

This document is both a usage guide and a technical walkthrough of how the
system works, hardware and software.

---

## 1. Overview

The player reads a `.wav` file off an SD card, parses its header, and streams
the raw audio samples to a DAC over the I2S protocol. A rotary encoder lets the
user browse and select songs; an OLED shows the song name, play state, and
playback progress. The whole system is written in MicroPython across a small set
of focused modules.

**Signal path at a glance:**

```
microSD (SPI) --> RP2040 (parse + buffer) --> I2S DAC --> amplifier/speaker
                          ^                       ^
                     si5351 clock ----------------+  (24.576 MHz master clock)
```

---

## 2. Hardware

### 2.1 Components

| Component        | Role                                                          |
| ---------------- | ------------------------------------------------------------- |
| YD-RP2040        | Microcontroller (RP2040, dual-core, running MicroPython)      |
| I2S DAC          | Converts the digital audio stream to analog line/headphone out |
| si5351           | Programmable clock generator; supplies the DAC's master clock |
| microSD card     | Audio storage (SPI interface, FAT32)                          |
| SSD1306 OLED     | 128x64 monochrome display (I2C)                               |
| Rotary encoder   | Browse / select, with integrated push-button                  |
| Mode button      | Toggles between control and song-select modes                 |

> Fill in exact part numbers for the DAC and si5351 board you used.

### 2.2 The three buses

A key design point worth highlighting: the three peripherals each sit on a
separate hardware bus, so they never contend for the same lines.

- **SD card** -> SPI0 (GPIO 0-3)
- **I2S DAC** -> I2S peripheral 0 (GPIO 6, 7, 9)
- **OLED**    -> I2C0 (GPIO 12, 13)

This separation matters: audio streaming hammers the SD SPI bus continuously,
and if the display shared that bus, every screen update would stall audio reads.
Keeping them independent is part of why playback stays clean.

### 2.3 Pin map

All assignments live in `pins.py`.

| Function                  | GPIO | Bus    |
| ------------------------- | ---- | ------ |
| I2S bit clock (SCK)       | 6    | I2S0   |
| I2S word select (WS/LRCK) | 7    | I2S0   |
| I2S master clock (MCLK)*  | 8    | (si5351) |
| I2S data (SD/SDIN)        | 9    | I2S0   |
| OLED SDA                  | 12   | I2C0   |
| OLED SCL                  | 13   | I2C0   |
| Encoder A                 | 18   | GPIO   |
| Encoder B                 | 17   | GPIO   |
| Encoder button            | 5    | GPIO   |
| Mode button               | 21   | GPIO   |
| SD MISO                   | 0    | SPI0   |
| SD CS                     | 1    | SPI0   |
| SD SCK                    | 2    | SPI0   |
| SD MOSI                   | 3    | SPI0   |
| SD card-detect            | 4    | GPIO   |

*The master clock is generated by the si5351, not the RP2040. The RP2040 only
drives SCK, WS, and data; the DAC takes its MCLK from the si5351.

### 2.4 The master clock relationship (the single most important detail)

The si5351 is configured to output **24.576 MHz**, which is exactly
**512 x 48000**. An I2S DAC that requires an external master clock will only
produce sound when MCLK is an exact integer multiple of the audio sample rate.
24.576 MHz divides cleanly by 48 kHz; it does **not** divide cleanly by 44.1 kHz
(24,576,000 / 44,100 = 557.3, not an integer). When the ratio is wrong, most
DACs simply mute rather than play noise.

This is why every audio file in this project must be 48 kHz (see Section 4). It
is the hardware constraint that drives the most important software rule.

---

## 3. Software architecture

The code is split into focused modules, each owning one responsibility.

| File           | Responsibility                                                |
| -------------- | ------------------------------------------------------------- |
| `main.py`      | The control loop; wires the modules together                  |
| `player.py`    | WAV streaming, buffering, and I2S output                       |
| `songs.py`     | Builds and navigates the song list from the SD card           |
| `sd_driver.py` | SD card SPI driver, mounting, file access, WAV header parsing |
| `encoder.py`   | Interrupt-driven rotary encoder + polled buttons              |
| `display.py`   | OLED screen layouts                                           |
| `pins.py`      | Central GPIO pin assignments                                  |
| `ssd1306.py`   | Low-level OLED hardware driver                                 |

### 3.1 Control flow

`main.py` runs a continuous loop that does the same few things every pass: read
the encoder, read the buttons, ask the player to top up its buffers, and refresh
the display when something has changed. It holds no audio data itself; it just
coordinates. The actual audio work happens in `player.py`.

### 3.2 The audio pipeline

The player **streams** audio rather than loading whole files into memory (an
11-minute song is ~110 MB of WAV data, far larger than the RP2040's RAM). The
pipeline has three stages:

1. **Read** - small chunks are read from the SD card into a ring of buffers.
2. **Buffer** - filled chunks wait in the ring; the I2S peripheral also has its
   own internal buffer (`IBUF_SIZE`) that the DMA engine drains continuously.
3. **Play** - the I2S peripheral clocks samples out to the DAC at exactly the
   file's sample rate.

The buffering between read and play is what absorbs timing variability: as long
as the next chunk arrives before the buffer empties, the listener hears
uninterrupted audio even though SD reads are bursty.

### 3.3 Two cores

Playback runs on the RP2040's second core (via `_thread`), so feeding audio
doesn't block the user interface running on the main core. (Note: MicroPython
uses a single interpreter lock, so the two cores share Python execution rather
than running fully in parallel; the split still helps because the audio work
overlaps with the interface's idle and I/O time.)

### 3.4 WAV header parsing

`sd_driver.read_wav_info()` reads the RIFF/WAVE header to extract sample rate,
bit depth, channel count, and the byte offset and length of the audio data. The
player uses these to configure the I2S peripheral to match the file and to know
where the samples begin and end.

### 3.5 Rotary encoder decoding

The encoder is read with **interrupts**, not polling. A polled approach misses
steps when the main loop is busy (e.g. during a display update), producing
erratic or wrong-direction counts. The interrupt fires on encoder edges and
updates a counter immediately, regardless of what the main loop is doing. The
direction is determined by comparing the A and B quadrature signals at each edge.

## 4. Audio file format

**Every file must be 48 kHz, 16-bit, stereo PCM WAV.**

This is a hard requirement, for the master-clock reason in Section 2.4: a file at
any other sample rate will not match the si5351's 24.576 MHz clock, and the DAC
will output silence with no error message. Convert any source audio with:

```
ffmpeg -i input.mp3 -ar 48000 -ac 2 -sample_fmt s16 output.wav
```

- `-ar 48000`     -> 48 kHz sample rate (matches the master clock)
- `-ac 2`         -> stereo
- `-sample_fmt s16` -> signed 16-bit PCM

Place WAV files at the **root** of the SD card. The song scanner ignores files
in subfolders and hidden files (anything beginning with `.`, such as the macOS
`.Trashes` and `.Spotlight-V100` folders that appear automatically).

## 5. Setup and use

### 5.1 Building it

1. Flash MicroPython onto the YD-RP2040 (use a firmware build matching the
   board's flash size so the full filesystem is available).
2. Format a microSD card as **FAT32** and copy 48 kHz WAV files to its root.
3. Open the board in VS Code with the MicroPico extension and upload all project
   files.
4. Insert the SD card and run `main.py` (or reset the board to autorun it).

### 5.2 Using it

The player has two modes, toggled with the **mode button**:

- **Control mode** (default): press the encoder button to play / pause. The
  display shows the song name, play state, and a progress bar with elapsed /
  total time.
- **Song-select mode**: turn the encoder to scroll the song list; press the
  encoder button to play the highlighted song (which also returns to control
  mode). Press the mode button to go back without selecting.

## 6. Engineering notes (selected problems solved)

These are the non-obvious technical challenges behind the working product, and
make good discussion points for a presentation.

### 6.1 Diagnosing silent playback

Early on, a correctly-formatted WAV produced no sound while a generated tone did.
The cause was the master-clock mismatch (Section 2.4): the files were 44.1 kHz
against a 48 kHz clock. The fix was standardizing all audio to 48 kHz.
**Lesson:** an I2S DAC with an external MCLK is silent, not noisy, when the clock
ratio is wrong - an easy symptom to misread.

### 6.2 Streaming vs. loading

Songs are far too large to fit in RAM, so the player reads them incrementally.
This shaped the entire architecture around buffering and read timing.

### 6.3 Memory pressure

The RP2040 has limited RAM, and the audio buffers consume a large share of it.
Oversized buffers caused out-of-memory errors when the I2S buffer couldn't find a
contiguous block. **Lesson:** bigger buffers are not automatically safer; they
compete with the heap, and on a fragmented heap a large contiguous allocation can
fail even when total free memory looks sufficient.

### 6.4 SD throughput and the timing budget

48 kHz / 16-bit / stereo audio consumes a fixed **187.5 KB/s** (48000 x 2 x 2
bytes). The SD card must sustain at least this rate continuously or the audio
buffer underruns and the sound stutters. Measured read throughput on the test
cards was around 290 KB/s, leaving roughly 1.5x headroom - enough, but not large,
which is why buffer sizing and avoiding bus contention mattered. Faster SD cards
have more margin.

### 6.5 Buffer sizing tradeoffs

Buffer sizes balance three competing pressures: larger buffers give more cushion
against read delays, but cost more memory and (for read chunk size) change
read efficiency. The working configuration was tuned empirically to the point
where playback was reliable without exhausting memory.

## 7. Known issues

- **Startup static (~100 ms):** a brief burst of static at the very start of a
  song, before the buffers fill, after which playback is clean. Several fixes
  were attempted (pre-filling buffers, padding with silence); each risked the
  stable mid-song playback, so it is currently left as a known cosmetic issue.
- **SD-card dependence:** playback reliability scales with SD read speed. Slow or
  worn cards reduce the timing margin and can cause stutter.

---

## 8. Possible future work

- **Volume control:** currently not implemented. Requires either a DAC with a
  digital volume register or scaling each sample in software (which adds CPU load
  to an already tight timing budget).
- **Eliminating the startup static** without disturbing steady-state playback.
- **Auto-advance** to the next song when one finishes.
- **Gapless or larger-library handling** as the song list grows.
